Contribute

We greatly value and encourage all kinds of contributions. Before getting started, please review the relevant section to ensure a smooth process for everyone and to assist the maintainers in managing contributions effectively. The community eagerly awaits your input!

If you love the project but don’t have the time to contribute, that’s absolutely fine! There are other simple ways to support and show your appreciation, which we’d greatly value:

  • Give the project a star
  • Share it on Twitter
  • Add a reference to it in your project’s README
  • Mention it at local meetups or share it with your friends and colleagues

I Have a Question

We assume you’ve gone through the available documentation before raising a question.

It’s recommended to search for existing issues before asking a question, as you may find helpful information. If you come across a relevant issue but need more clarification, feel free to add your question there. Additionally, consider reaching out to the community on the RUMBLESHEET.JS Discord server for further support.

If you still need clarification after searching, we suggest the following:

  • Open a new issue.
  • Include as much detail as possible about the issue you’re encountering.
  • Provide relevant information such as project version, browser, and operating system.

We (or someone from the community) will address the issue as soon as possible.

Prior to Reporting a Bug

To ensure efficient bug resolution, a well-documented report is crucial. We kindly request you to investigate thoroughly, gather all necessary details, and provide a clear description of the issue. Completing the following steps in advance will help us address the problem as quickly as possible.

  • Ensure you are using the latest version of the library.
  • Confirm that the issue is not caused by a misconfiguration or incompatible environment setup (e.g., conflicting Node.js, browser, or framework versions). Be sure to review the documentation thoroughly. If you need assistance, check the this section for guidance.
  • Look through the library's issue tracker to see if the problem has already been reported or resolved by others.
  • Search the web, including forums like Stack Overflow, for discussions or solutions related to the issue.
  • Collect detailed information about the issue, including:
    • Any error messages or stack traces from the browser console or Node.js runtime
    • The operating system, platform, and environment (e.g., Windows, macOS, Linux, browser type/version, Node.js version)
    • The version of the library and any dependencies being used
    • The code snippet or input that triggers the issue, along with the observed output
  • Test whether the issue is consistently reproducible and whether it occurs in previous versions of the library.

Guidelines for Writing an Effective Bug Report

Do not report security-related issues, vulnerabilities, or bugs with sensitive information on the issue tracker or any public platforms. Instead, please email these types of bugs to opensource@pandevs.org.

We use GitHub issues to track bugs and errors in the project. If you encounter an issue with the library:

  • Open a new issue (at this stage, please avoid labeling it as a bug until we confirm).
  • Describe the expected behavior and the actual behavior.
  • Provide as much context as possible, including clear steps to reproduce the issue. This typically involves sharing relevant code. For effective bug reports, isolate the issue and create a minimal test case.
  • Include the information you've gathered from the previous section.

Once the issue is submitted:

  • The project team will label the issue appropriately.
  • A team member will attempt to reproduce the issue using the steps you provided. If there are no steps or it's unclear how to reproduce the issue, the team will request those steps and mark the issue as needs-repro Issues marked with needs-repro will not be addressed until they can be reproduced.
  • If the team successfully reproduces the issue, it will be labeled needs-fix and potentially other tags (e.g., critical). The issue will then be assigned for resolution.
  • Please ensure you use the provided issue templates.
